## 输出规则

### 优先级排序（严格按此顺序输出）

#### 1. 阻塞问题追踪（最重要，预算 70%）

对每一对相邻轮次 (R1→R2, R2→R3, ...)，**逐条语义匹配** blocking_issues：
- 描述不同但指向同一根因 → 视为同一问题
- 从 blocking_issues 消失 → 标记为已修复
- 之前不存在、本轮首次出现 → 标记为新引入

按以下格式输出：

```
### ❌ 仍存在（跨轮未解决）
- [R1→R5] API 接口契约缺失 — 共 5 轮反复出现
  * 原因分析：Builder 仅添加了候选 API 名称，未指定具体端点 URL 和响应字段映射

### ✅ 已修复
- [R1→R3] Django 版本号事实错误 — R4 起 blocking_issues 中消失

### ⚠️ 本轮新引入
- [R5] LWW 物理时间戳与 Yjs CRDT 矛盾 — 本轮首次出现
```

**严格约束**：
- 如果 blocking_issues 为空 []，对应分类下输出「（无）」而非凭空推测
- 「仍存在」条目必须附原因分析（为什么 Builder 没能修好），帮助 Builder 调整修复方向
- 语义匹配优先于字面匹配——如 "API 端点未指定" 和 "接口契约缺失" 视为同一问题

#### 2. 得分趋势（预算 20%，极度精简）

标记 ⚠️ 退化：任何维度相比上一轮降低 ≥ 3 分：

